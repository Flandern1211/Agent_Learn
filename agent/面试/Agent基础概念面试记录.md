# Agent 基础概念面试记录

> 面试时间：2026-05-19
> 考察方向：Agent 核心概念、架构分层、编排模式、记忆机制

---

## 问题一：什么是 Agent？与大模型有什么本质不同？

### 参考答案

Agent = LLM + Harness，即大模型加上为了发挥其能力和实用性而发展出的一切工程体系。

与大模型的本质区别在于 Harness 工程。传统大模型一般只有单会话问答，没有工具调用能力，没有与外界进行连接交互的能力。

### 评分与反馈

- **方向正确**，"LLM + Harness"这个框架是合理的
- 需要更结构化地展开 Harness 的具体组成
- 注意表述准确性（如"没有工具调用能力"需限定为"没有自主工具调用能力"）

---

## 问题二：Harness 具体包含哪些核心组成部分？

### 参考答案

分为四个层面：

| 层级 | 关注点 | 具体内容 |
|---|---|---|
| **能力层** | Agent 能做什么 | LLM 本身的推理、生成、理解能力 |
| **连接层** | Agent 如何与外界对话 | MCP 协议、外部 API、工具调用协议（Function Call） |
| **构建层** | Agent 怎么被搭起来 | Prompt 工程、Agent SDK、Agent 框架、任务编排逻辑（ReAct、Plan-and-Execute） |
| **运行管控层** | Agent 怎么长期稳定运行 | 状态管理、记忆机制、执行环境、可观测性 |

### 评分与反馈

- 四层结构有逻辑，说明有结构化思考能力
- **不足**：能力层和连接层边界模糊，MCP 同时出现在两层中
- **改进建议**：能力层只关注模型本身能力，工具相关统一归入连接层

## 问题三：Agent 的工具调用与传统大模型的工具调用本质区别是什么？

### 参考答案

核心区别在于**自主性**。Agent 的工具调用是模型自主决策的，体现在四个环节：

1. **决定要不要调用** —— 不是每次都调
2. **决定调用哪个工具** —— 从工具列表中选择
3. **决定传什么参数** —— 从自然语言中提取构造
4. **决定怎么用返回结果** —— 整合到推理链中

其中"决定要不要调用"和"决定调用哪个"对模型能力要求最高，因为这需要模型理解任务语义并匹配工具描述。

### 评分与反馈

- 抓住了"自主性"这个关键点
- **不足**：只说了一半，需要展开四个环节
- 对参数构造的难度有正确认知（模糊 prompt 下的参数提取是难点）

## 问题四：MCP 在 Agent 架构中解决什么问题？属于哪一层？

### 参考答案

MCP 属于**连接层**。它解决的核心问题是：

- **标准化**：统一了 Agent 与外部工具/数据源的交互协议
- **解耦**：Agent 不需要为每个工具写专门的集成代码
- **可发现性**：工具可以通过 MCP 协议自描述，Agent 可以动态发现可用工具

虽然 MCP 最终目的是让模型更好地使用工具（能力层的效果），但 MCP 协议本身是连接层的基础设施。

### 评分与反馈

- 正确归入连接层
- **不足**：回答偏简略，可以补充 MCP 解决的具体问题（标准化、解耦等）

## 问题五：ReAct 模式是什么？有什么局限性？

### 参考答案

ReAct 是最常见的 Agent 编排模式，核心循环：

```
思考 (Thought) → 行动 (Action/Tool Call) → 观察 (Observation) → 思考...
```

循环持续到模型认为不再需要工具调用，给出最终结果。

**局限性**：对于复杂多步骤任务（如"写一个完整的 Go HTTP 服务器，包含路由、中间件、数据库连接"），ReAct 可能：
- 缺乏全局规划，导致模块之间边界混乱
- 把本应严格区分的部分混在一起实现
- 容易陷入局部最优，缺少全局视角

### 评分与反馈

- 能正确描述 ReAct 循环机制
- 能识别出 ReAct 在复杂任务上的局限性
- **不足**：回答过于简略，需要更详细地展开

## 问题六：Plan-and-Execute 模式与 ReAct 的核心区别？

### 参考答案

| 维度 | ReAct | Plan-and-Execute |
|---|---|---|
| 思路 | 想一步做一步 | 先想好全部再做 |
| 循环 | 思考→行动→检查→思考... | 规划→执行步骤1→执行步骤2→... |
| 适合场景 | 简单任务、探索性任务 | 复杂多步骤任务 |
| 缺点 | 缺乏全局规划 | 计划出错时回溯困难 |

**Plan-and-Execute 的缺点**：如果执行到第 3 步发现第 1 步的计划就是错的，需要动态重新规划（Dynamic Replanning）。这是该模式的核心挑战。

**进阶关键词**：Replanning、Dynamic Replanning、LangGraph 的 Plan-and-Execute 实现

### 评分与反馈

- 知道 Plan-and-Execute 的基本概念
- **不足**：对缺点和回溯机制不了解，需要深入学习

## 问题七：Agent 中的短期记忆和长期记忆分别是什么？

### 参考答案

**通俗解释**：

- **短期记忆** = 助手手边的便签纸。记录"刚才聊了什么"——当前对话上下文、用户指令、中间结果。对话结束就丢弃。
- **长期记忆** = 助手的笔记本。记录"这个用户是谁、喜欢什么、之前做过什么"。跨对话持久保存。

**技术对应**：

| 类型 | 实现 | 内容 | 生命周期 |
|---|---|---|---|
| 短期记忆 | LLM 的 Context Window | 当前对话的 prompt、历史消息、工具返回结果 | 单次对话 |
| 长期记忆 | 向量数据库（如 ChromaDB）或结构化存储 | 经过 embedding 的历史信息，需要时检索塞回 context window | 跨对话持久 |

### 评分与反馈

- 诚实表示不了解，比硬编更好
- **建议补强**：学习 RAG + 向量数据库的完整流程，了解记忆的 CRUD 和检索策略

## 问题八：如果通过 MCP 提供 memory 工具，你会设计哪些 tool？

### 参考答案

基础三个工具：

1. **extract_memory** —— 根据上下文提取重要内容
2. **store_memory** —— 将重要内容转化为向量格式并存入数据库
3. **query_memory** —— 根据问题的向量格式查询数据库

需要补充的工具：

4. **update_memory** —— 更新已有记忆（如用户偏好变更）
5. **delete_memory** —— 删除过时或错误的记忆，防止记忆污染

还需要考虑的问题：

- **记忆类型区分**：偏好、事实、事件需要不同的存储和检索策略
- **元数据管理**：时间戳、来源对话、置信度等
- **记忆冲突处理**：新旧信息矛盾时如何决策

### 评分与反馈

- 三个基础工具覆盖了"存"和"查"
- **不足**：缺少更新、删除、防污染等考量
- 这是一个开放性问题，重点考察系统设计思维
